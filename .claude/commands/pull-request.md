현재 브랜치의 변경사항을 분석하고 리뷰어가 이해하기 쉬운 Pull Request를 생성하거나, 이미 열려 있는 PR의 본문을 최신 커밋에 맞춰 갱신한다.

> **베이스 브랜치**: 프로젝트는 git-flow를 따르므로 feature PR의 base는 **`develop`**이다. 상세 규칙은 `.claude/rules/git-flow.md` 참조.

## 절차

1. **브랜치 상태 확인**
   - `git branch --show-current`로 현재 브랜치 확인 (main/master/develop면 중단하고 알림)
   - `git status`로 working tree clean 확인 (uncommitted 변경 있으면 먼저 커밋하도록 안내)
   - `git fetch origin develop`으로 베이스 브랜치 최신화

2. **기존 PR 존재 여부 확인** ⭐
   - `gh pr view --json number,url,title,body,state --jq '{number,url,title,state}'`로 현재 브랜치에 연결된 PR 조회
   - 결과에 따라 분기:
     - **PR 없음** → 신규 생성 플로우 (아래 3~7단계)
     - **PR 있음 (OPEN)** → 업데이트 플로우 (아래 8단계)
     - **PR 있음 (MERGED/CLOSED)** → 사용자에게 알리고 신규 생성 여부 확인

3. **변경 내용 수집**
   - `git log --oneline develop..HEAD`로 이 브랜치의 전체 커밋 히스토리 확인
   - `git diff develop...HEAD --stat`로 변경 파일 개요 파악
   - `git diff develop...HEAD`로 실제 변경 내용 분석
   - ⚠️ 최신 커밋 하나가 아니라 **브랜치에 포함된 모든 커밋**을 분석해야 함

4. **원격 브랜치 동기화**
   - 원격에 브랜치가 없으면 `git push -u origin HEAD`
   - 원격에 있으면 로컬이 앞서 있는지 확인 후 필요 시 `git push`

5. **PR 제목 작성** (신규 생성 시)
   - 형식: `prefix: 한글메시지` (약 70자 이내)
   - prefix는 `.claude/rules/git-commit.md`의 커밋 prefix 규칙을 그대로 따름
   - 여러 종류의 변경이 섞였다면 가장 핵심적인 변경을 대표로 선택
   - 커밋 메시지 단순 나열이 아닌, **PR 전체를 관통하는 주제**로 작성

6. **PR 본문 작성** (아래 템플릿 사용)

7. **PR 생성**: `gh pr create --base develop --title "..." --body "$(cat <<'EOF' ... EOF)"`

8. **기존 PR 업데이트** (기존 PR이 OPEN일 때)
   - 신규 커밋이 PR 본문에 반영되도록 본문을 재작성 (변경 그룹 추가/수정/삭제)
   - 제목은 기본적으로 유지. PR의 주제 자체가 바뀐 경우에만 제목도 수정 (사용자에게 먼저 확인)
   - 업데이트 명령: `gh pr edit <번호> --body "$(cat <<'EOF' ... EOF)"`
   - 본문에 **"왜 이 커밋이 추가됐는지"** 배경을 포함해 리뷰어가 히스토리를 이해할 수 있게 한다
   - 이전 설명과 달라진 부분(예: 커스텀 구현 → 패키지 사용)은 **리뷰 포인트** 블록으로 강조

9. **결과**: 신규 생성/업데이트된 PR URL을 사용자에게 전달

## PR 본문 템플릿

```markdown
## 개요

<이 PR이 무엇을, 왜 하는지 1~3줄로 요약. "무엇"보다 "왜"에 집중>

## 주요 변경사항

<리뷰어가 가장 먼저 봐야 할 변경을 논리 단위로 그룹화. 파일명 나열이 아닌 "무엇이 어떻게 바뀌었는지" 서술>

### <그룹 제목 1 — 예: 로그인 화면 UI 구현>

- 핵심 변경점 1 (관련 파일 경로 언급)
- 핵심 변경점 2
- 결정 배경이나 트레이드오프가 있다면 한 줄 설명

### <그룹 제목 2 — 예: 디자인 시스템 배경색 변경>

- ...

## 스크린샷 / 데모

<UI 변경이 있으면 before/after 캡처나 영상 링크. 없으면 "해당 없음">

## 테스트 방법

<리뷰어가 직접 확인할 수 있는 구체적인 단계>

- [ ] `flutter pub get` 후 `flutter run`
- [ ] <확인할 동작 1>
- [ ] <확인할 동작 2>

## 관련 문서 / 이슈

<참고한 스펙 문서, 와이어프레임, 이슈 번호 등. 없으면 생략>

---

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

## 본문 작성 규칙

### 리뷰어 가독성 최우선

- **파일 목록 나열 금지**: "`lib/features/auth/view/login_screen.dart` 수정" ❌
  → "로그인 화면에 Google 로그인 버튼 추가" ⭕
- **변경을 논리 단위로 그룹화**: 같은 의도의 변경은 한 섹션에 묶어 설명
- **"왜"를 설명**: 단순히 "무엇을 바꿨다"가 아니라 "왜 그렇게 바꿨는지" 배경 포함
- **용어 통일**: 스펙 문서(`.claude/rules/`, `.claude/ui/`)에서 쓰는 용어를 그대로 사용

### 길이 가이드

- 개요: 1~3줄 (한 문단)
- 주요 변경사항: 그룹당 3~6줄, 총 3그룹 이내 권장
- 테스트 방법: 체크박스 3~6개 정도
- 너무 길어지면 리뷰어가 읽지 않음 → **핵심만**

### 마크다운 활용

- 변경 그룹 구분은 `###` 사용
- 중요한 키워드는 **굵게**
- 파일 경로나 코드 식별자는 `` `백틱` ``으로 감싸기
- 긴 diff 인용은 피함 (필요 시 짧게만)

## 기존 PR 업데이트 규칙

PR이 이미 열려 있는 상태에서 `/pull-request`가 다시 호출되면, **항상 본문을 최신 상태로 재작성**한다. 단순히 새 커밋만 덧붙이는 것이 아니라 PR 전체를 다시 읽었을 때 자연스럽도록 재구성한다.

### 재작성 방침

- **덧붙이기 금지**: "추가 커밋" 같은 새 섹션을 맨 아래 append하지 말 것. 해당 변경을 **기존 "주요 변경사항"의 적절한 그룹에 녹여 넣는다**.
- **변경된 의도 반영**: 이전 PR 본문의 설명과 실제 코드가 어긋나면(예: 초기에는 커스텀 구현 → 후속 커밋에서 패키지 사용으로 전환) 본문의 해당 부분을 현재 코드 기준으로 다시 쓴다.
- **리뷰 포인트 블록 추가**: 설계 결정이 바뀐 경우 `> **리뷰 포인트**:` 인용 블록으로 배경을 한두 문장 남겨 리뷰어가 히스토리를 이해할 수 있게 한다.
- **테스트 방법 동기화**: 새 커밋이 새로운 의존성/명령을 추가했다면 체크리스트에 반영 (예: `flutter pub get` → `sign_in_button` 설치 확인).
- **관련 문서 링크 추가**: 새로 참조한 외부 패키지, 문서, 이슈가 있으면 "관련 문서" 섹션에 추가.
- **제목 보존**: 제목은 기본적으로 바꾸지 않는다. PR 주제 자체가 달라진 경우(예: `feat` → `refactor`)에만 사용자에게 확인 후 변경.

### 실행 커맨드

```bash
# 1. 현재 PR 번호 확인
PR_NUMBER=$(gh pr view --json number --jq '.number')

# 2. 본문 재작성 후 업데이트
gh pr edit "$PR_NUMBER" --body "$(cat <<'EOF'
<재작성된 본문>
EOF
)"
```

## 규칙

- PR 제목/본문 모두 **한글**로 작성 (파일 경로, 코드 식별자 제외)
- prefix는 영어 소문자
- 베이스 브랜치는 기본적으로 **`develop`** (git-flow). `main`은 `/release` 커맨드 전용
- PR 생성/업데이트 전 반드시 `git push`로 원격 동기화
- 본문은 반드시 HEREDOC으로 전달해 이스케이프 문제 방지
- main/master/develop 브랜치에서는 PR 생성하지 않음 (feature 브랜치에서만)
- `gh pr create --draft` 옵션은 사용자가 명시적으로 요청할 때만 사용
- 생성/업데이트 후 PR URL을 반환해 사용자가 바로 확인할 수 있게 함
- 이미 MERGED/CLOSED된 PR은 자동 업데이트하지 않고 사용자에게 신규 생성 여부를 먼저 확인
