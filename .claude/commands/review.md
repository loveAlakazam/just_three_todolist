PR 번호를 인자로 받아 코드 리뷰를 수행하고, 결과를 GitHub PR 코멘트로 게시한다.

> **인자**: `#<PR번호>` 또는 숫자만. 생략 시 현재 브랜치의 열린 PR을 자동 감지한다.

## 절차

1. **PR 확인**
   - 인자가 없으면 `gh pr view --json number,url,title,state`로 현재 브랜치 PR 조회
   - PR이 MERGED / CLOSED 상태면 리뷰 중단하고 사용자에게 알림
   - `gh pr view <number>`로 PR 제목, 본문 확인
   - `gh pr diff <number>`로 전체 diff 수집

2. **변경 분석**
   - diff를 논리 단위(기능/레이어/파일 그룹)로 분류
   - 아키텍처 규칙 (`.claude/rules/architecture.md`) 준수 여부 확인
   - MVVM 레이어 분리 원칙 위반 탐지 (View에 비즈니스 로직, Repository에 UI 참조 등)

3. **리뷰 수행** — 아래 항목을 체계적으로 검토

   ### 버그 / 정확성
   - 런타임 예외 발생 가능 코드
   - 비동기 타이밍 문제 (race condition, state가 영구 로딩 상태로 남는 경우 등)
   - 잘못된 null 처리

   ### 보안
   - 민감 정보 노출 (키, 토큰이 로그/번들에 포함되는지)
   - Supabase RLS 정책 누락 또는 과도한 허용
   - 인증 가드 우회 가능 경로

   ### 코드 품질
   - 데드 코드 (정의했으나 호출되지 않는 메서드)
   - 문자열 기반 분기 (타입 안전하지 않은 조건)
   - silent catch (에러를 삼키는 빈 catch 블록)
   - UI 상태 미완성 (로딩/에러 상태가 화면에 반영되지 않음)

   ### DB / 마이그레이션
   - `updated_at` 자동 갱신 트리거 누락
   - RLS 정책 완전성 (SELECT / INSERT / UPDATE / DELETE 각각 검토)
   - 인덱스 누락

   ### Flutter / Dart 관례
   - `onPressed: null` vs `onPressed: () {}` — null이 시각적 비활성화를 제공함
   - `ref.listen`을 `build()` 안에서 사용하는 올바른 패턴
   - `StatefulShellRoute` 탭 전환 방식

4. **우선순위 분류**
   - **즉시 수정**: 실제 버그, 보안 취약점, 데이터 손실 가능성
   - **머지 전 권고**: UX 문제, 기능 누락, DB 스키마 미완성
   - **다음 PR 수용 가능**: 리팩터링, 코드 스타일, 낮은 영향도 개선
   - **잘 된 부분**: 긍정적인 구현 사항 명시

5. **GitHub PR 코멘트 게시** ⭐
   - 리뷰 결과를 아래 템플릿으로 정리해 `gh pr comment <number> --body "..."` 로 게시
   - 반드시 HEREDOC으로 전달해 이스케이프 문제 방지
   - 코멘트 게시 후 URL을 사용자에게 반환

6. **결과 반환**
   - 게시된 코멘트 URL 출력
   - 즉시 수정이 필요한 항목만 요약해서 사용자에게 안내

## 코멘트 게시 명령어

```bash
gh pr comment <number> --body "$(cat <<'EOF'
<리뷰 본문>
EOF
)"
```

---

## 규칙

- 리뷰 본문과 GitHub 코멘트 내용은 동일해야 한다 (별도 요약 금지)
- MERGED / CLOSED PR은 리뷰하지 않는다
- 파일 경로, 라인 번호를 최대한 구체적으로 명시한다
- 긍정적인 부분도 반드시 포함한다 — 지적만 하는 리뷰는 금지
- 코멘트는 한글로 작성한다 (파일 경로, 코드 식별자, 패키지명 제외)
- 즉시 수정 항목이 없으면 해당 섹션을 생략하고 "이슈 없음" 대신 "잘 된 부분"으로 긍정적으로 마무리
